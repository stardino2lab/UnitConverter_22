## §3 Mom Test 인터뷰 기록

### 질문 1

> **마지막으로 meter, feet, yard 중 하나를 다른 단위로 바꿔야 했을 때**, 그게 **어떤 일** 때문에 필요했고, **그때 실제로 어떻게** 처리했는지 **처음부터 끝까지** 말해 주세요.

### 답변 (페르소나1 · 시뮬레이션)

- 지난주 해외 건축 도면 PDF — 치수 feet·inch
- 내부 보고서 meter → 엑셀 `× 0.3048`
- yard 미사용, feet→meter만
- 반복 복붙·수식 드래그
- inch→feet 합치기 누락 → 숫자 오류 → 동료 검출
- **~40분**, 구글 `feet to meter` 검산

### Mom Test 평가

| 기준 | 평가 |
|------|------|
| 사실 (과거 행동) | ✅ 구체적 사건 |
| 구체성 | ✅ 도구·시간·실수 |
| 솔루션 혼입 | ✅ 없음 (엑셀·검색 = 기존 행동) |
| Good Data | ✅ 40분, 1회 오류 |

## §4 Mom Test 분석

### 표면 문제 (솔루션 혼입 — PRD와 구분)

- “CLI로 meter/feet/yard 3단위 일괄 변환”
- “JSON/CSV/table 출력”
- “`meter:2.5` 파싱 CLI”

→ Mom Test에서 **아직 증명되지 않음** (페르소나1은 엑셀·검색 사용).

### 진짜 문제 (1문장)

> **해외 도면(feet/inch 혼재)을 내부 meter 보고서로 옮길 때, 스프레드시트 수식·수동 검산에 시간이 들고, 단위 조합·반복 입력 실수로 숫자가 틀릴 수 있다.**

### Mom Test 증거 3줄

1. **40분** — 반복 수식·검산이 실제 비용.
2. **inch→feet 누락** — 단위 **조합** 실수가 핵심 pain (단순 3단위 변환기 아님).
3. **엑셀+구글** — CLI·3단위 동시 출력 요구와 **사용 습관 간격** 존재.
